# MULTI-AGENT COORDINATION — 多 Agent 协调协议

> **四层架构 + 审查协调分离 + 问题部门间闭环。**

---

## 架构概览

```
┌─────────────────────────────────────────────────┐
│                老板 (Coordinator)                 │
│         拆解任务 → 委派部门 → 汇总结果             │
└──────┬──────────┬──────────┬────────────────────┘
       │          │          │
       ▼          ▼          ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│  研发部   │ │  运维部   │ │  测试部   │  ... 情报部
└────┬─────┘ └────┬─────┘ └────┬─────┘
     │            │            │
     └─────┬──────┴──────┬─────┘
           │  横向协调     │
           ▼              ▼
    ┌─────────────┐  ┌──────────────┐
    │  审核员      │  │  调度员       │
    │  质量判定     │  │  部门间协调    │
    │  只判对错     │  │  不上报老板    │
    └─────────────┘  └──────────────┘
```

---

## 四层角色定义

| 层级 | 角色 | 职责 | 不能做的事 |
|------|------|------|-----------|
| **老板** | Coordinator | 拆解任务、委派部门、汇总结果 | 不能干涉部门技术方案（除非违反约束） |
| **部门** | Departments | 执行具体工作 | 不能跨部门指挥其他部门 |
| **审核员** | Reviewer | 对照目标做二元质量判定 | 不能归因、不建议修复方案、不评价部门 |
| **调度员** | Dispatcher | 横向归因、制定返工计划 | 不能改质量标准、不能改任务目标 |

---

## 核心设计原则

### 1. 问题在部门间闭环

调度员是**横向协调层**——部门间的问题由调度员协调解决，**不上报老板**。只有调度员判断"超出部门能力范围"时才升级。

### 2. 审核员只管质量

审核员对照**原始任务目标**做二元判定（pass/fail），不管是谁做的、怎么做。面对跨部门矛盾时，只标记为 `cross_dept` 而不选边站——归因是调度员的活。

### 3. 老板控盘协调循环

主 Agent 始终控制流程，不交给子 Agent 自主决策流转：

```
委派部门 → 提取报告 → 委派审核员 → 解析判定
    → (fail) 委派调度员 → 分配返工 → 委派部门返工
    → 再次审核 → (pass) 完成
    → 最多 3 次重试 → 超出则 failed
```

## 审核员判定标准（三级）

| 标准 | 适用场景 | 判定方法 |
|------|---------|---------|
| **strict** | 接口契约、API 定义、文件格式 | 精确字符串/结构匹配 |
| **practical** | 功能验收、业务逻辑 | 功能等效即可 |
| **existence** | 交付物检查 | 文件存在且非空 |

### 跨部门矛盾处理

当部门 A 说"交付了 X"，部门 B 说"找不到 X"时：

- ✅ **标记为 `cross_dept`** — 描述矛盾现象
- ❌ **不单方面判任一方失败** — 可能是路径约定不清晰
- ❌ **不建议修复方案** — 归因是调度员的工作

---

## 调度员归因框架

### 失败类型分类

| 类型 | 示例 | 归因策略 |
|------|------|---------|
| 单部门缺陷 | 端口号写错 | 该部门返工 |
| 接口不匹配 | JSON 格式与预期不同 | 检查 Coordinator 的接口约定是否清晰 |
| 路径不一致 | 文件写入路径与查找路径不同 | 优先检查 Coordinator 是否给了绝对路径 |
| 交付物缺失 | 声称已交付但找不到 | 先自己 `search_files` 验证 |
| 理解偏差 | 实现了 REST 而非 gRPC | 回看 Coordinator 指令清晰度 |

### 升级条件

以下情况升级到 Coordinator（告知而非交给老板）：

1. 接口约定本身模糊，两个部门都没有明显错误
2. 3 次返工后仍失败
3. 部门产出方向性错误（超出修 bug 范畴）
4. Coordinator 的委派指令是问题根源

## 异常处理清单

### E1: 部门报告"目录为空"

**症状**：部门说完成了但找不到交付物
**根因**：子 Agent 文件写入路径与预期不同
**处理**：
1. 检查部门报告中的实际路径
2. 在自己的环境中用 `search_files` 搜索
3. 确实不存在 → 标记为 E1 → 审核时重点标注

### E2: 跨部门路径不一致

**症状**：dev 说文件在 `/path/a/`，ops 说找不到
**处理**：
1. 这是调度问题信号，不是单方面错误
2. 审核时标注"跨部门路径不一致"
3. 让调度员归因——可能是接口定义不清

### E3: File-mutation verifier 假阳性

**症状**：子 Agent 声称已修改文件，但 `cat` 发现没变
**处理**：以实际 `read_file` 结果为唯一真相源

### E4: 审核员与部门结论矛盾

**症状**：部门认为完成了，审核员判定失败
**处理**：审核员判定是权威的，直接进入调度+返工流程

---

## 产前验证（Phase 2.5）

在委派部门之前强制执行：

1. **路径可及性**：确认工作目录存在且可写
2. **依赖文件就绪**：初始文件用 `read_file` 验证
3. **跨部门路径一致性**：所有委派指令使用相同的绝对路径
4. **预检清单**：每个部门需要的外部信息已准备好

```json

  "task_id": "uuid",

  "to": "dev-team",

    "context": "工作目录: /absolute/path/ ...",
    "deliverables": ["server.py", "requirements.txt"]
  }
}
```
