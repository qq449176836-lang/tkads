# METHODOLOGY — 思考逻辑与方法论

> **Agent 面对问题的思考框架：如何检索、如何拆解、如何执行、如何闭环。**

---

## 1. 三层检索法

> 遇到问题时，从精确到模糊逐层寻找答案，避免从零推理。

```
Layer 1: 精确匹配
  → 用 search_files / grep 搜索项目内已有代码、配置、文档
  → 检查 Memory 中是否有相同问题的记录
  → 查看已安装 Skill 库是否有直接匹配

Layer 2: LLM 语义扩展
  → 将问题的关键词做语义扩展（同义词、相关领域）
  → 扩大检索范围到类似项目、相近技术栈

Layer 3: 遍历 Skill 库对比
  → 逐个加载相关 Skill，对比其方法论是否可迁移
  → 即使不完全匹配，核心思路可能复用

原则：能用现成方案就不造轮子，能搜索就不推理，能参考就不原创。
```

## 2. DAG 分解法

> 大目标 → 子任务 DAG → 逐层并行执行。

### 分解流程

```
大目标输入
    │
    ▼
┌─────────────────────────────────────┐
│ Step 1: 识别独立子目标               │
│   → 哪些任务可以并行？（无依赖）       │
│   → 哪些任务必须串行？（有依赖）       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 2: 构建 DAG                     │
│   A → B → D                         │
│   A → C → D    （A 是 B/C 的前置）    │
│   E → F        （E/F 独立链路）       │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 3: 逐层并行执行                 │
│   Wave 1: A, E  (无依赖，并行)       │
│   Wave 2: B, C, F (依赖上一波完成)   │
│   Wave 3: D (依赖 B+C 完成)          │
└─────────────────────────────────────┘
```

### 并行工具选择

| 工具 | 适用场景 |
|------|---------|
| `delegate_task` (batch) | 重度推理任务（代码审查、研究分析） |
| `terminal(background=true)` | 独立 CLI 操作（下载、构建、安装） |
| `write_file` (多次调用) | 创建多个独立文件 |

### 反模式

- ❌ 串行执行所有无依赖步骤
- ❌ 过度拆分导致子 Agent 上下文孤立，互相找不到产出
- ❌ 并行后不验证结果就进入下一步

## 3. 棘轮循环（Ratchet Cycle）

> **"经验不丢失、知识不退化"** 的闭环机制。

```
        遭遇问题
           │
           ▼
    ┌──────────────┐
    │  1. 诊断      │ ← 精准定位根因（不修表层症状）
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  2. 修复      │ ← 最小化改动，避免连锁影响
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  3. 记录      │ ← 写入 Memory（持久事实）
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  4. 提炼      │ ← 从具体案例提取通用模式
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  5. 标准化    │ ← 生成 Skill / 脚本 / 文档模板
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  6. 推送      │ ← 上传 GitHub / 更新 Skill 库
    └──────────────┘
           │
           └──→ 下次遇到同类问题自动加载 → 循环
```

### Memory vs Skill 的写入门槛

| 知识类型 | 存放位置 | 触发条件 |
|---------|---------|---------|
| 一次性事实 | Memory | 任意有价值的发现 |
| 重复模式（>=3 次） | Skill | 高频出现、可标准化 |
| 持久约定 | Memory / Constitution | 跨会话必须保持的规则 |

## 4. 验证先行原则

> **先验证可行性，再大规模执行。不走弯路，不做完反复修改。**

### 验证清单

| 阶段 | 验证动作 | 工具 |
|------|---------|------|
| **方案验证** | 最小可行实验（MVE） | 单命令测试 |
| **创建验证** | 文件/服务是否真的产生 | `stat` / `read_file` / `curl` |
| **集成验证** | 新旧代码是否兼容 | 端到端测试 |
| **部署验证** | 服务是否真的用新代码运行 | `netstat` 确认 PID 单一 |

### 子 Agent 结果验证

> 子 Agent 的 self-report 不可信。

子 Agent 声称"上传成功"、"文件已写入"时，必须获取可验证的句柄（URL、绝对路径、HTTP 状态码）并主动验证。

## 5. 多 Agent 协调模式

### 四层架构

```
老板 (Coordinator)     ← 拆解任务、委派、汇总
    │
    ├── 研发部 (dev)   ← 代码编写、架构设计
    ├── 运维部 (ops)   ← 部署、环境配置
    ├── 测试部 (test)  ← 质量验证、回归测试
    └── 情报部 (intel) ← 信息收集、技术调研
    │
    ├── 审核员 (Reviewer)  ← 质量判定（只判对错，不管谁做）
    └── 调度员 (Dispatcher) ← 部门间横向协调（不上报老板）
```

### 关键原则

1. **问题在部门间闭环** — 调度员协调解决部门间问题，不升级到老板
2. **审核员只管质量** — 二元判定 pass/fail，不归因
3. **老板控盘** — 主 Agent 始终控制流程，不交给子 Agent 自主决策流转

### 协调循环

```
委派部门 → 提取报告 → 委派审核员 → 解析判定
    → (fail) 委派调度员 → 分配返工 → 委派部门返工
    → 再次审核 → (pass) 完成
    → 最多 3 次重试 → 超出则 failed
```

## 6. 决策树：何时用哪个工具

```
需要执行代码？
  ├── 简单脚本 → terminal (foreground)
  ├── 长时间任务 (>10min) → terminal(bg + notify)
  └── 交互式 CLI → terminal(pty=true)

需要推理分析？
  ├── 独立可并行 → delegate_task(batch)
  ├── 需要用户交互 → 不能 delegate
  └── 需要访问特定工具 → delegate_task(toolsets=[...])

需要查询信息？
  ├── 项目内 → search_files / read_file
  ├── 公开 Web → web_extract / curl
  └── 交互式页面 → browser_navigate + browser_snapshot
```
