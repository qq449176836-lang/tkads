## 二十三、系统思维

让整个体系更坚固、更抗风险的底层设计原则：

### 23.1 冗余设计
```
关键路径不要单点依赖。

例子:
  - 数据有 JSON 文件 + 数据库双重存储
  - 配置有 .env + 模板文件双备份
  - 采集有 cron + 手动触发双通道
  - 仓库有 GitHub + 本地双备份
```

### 23.2 优雅降级
```
出问题时不能全挂。

设计原则:
  - VikingDB 连不上 → 降级到本地存储
  - 数据库连不上 → 用 JSON 文件支撑
  - 网络不通 → 缓存最近结果
  - 某个模块挂了 → 其他模块不受影响
```

### 23.3 反脆弱
```
系统要从波动和混乱中获益。

体现:
  - 每次报错都让系统变强（ENP循环）
  - 每次用户纠正都让对话更精准
  - 每次故障都变成本地知识的一部分
  - 越用越好用，越错越聪明
```

### 23.4 反馈回路
```
每个动作都要有反馈信号。

原则:
  - 脚本跑完要有退出码验证
  - API 调用要有响应解析
  - 写操作要有读操作验证
  - 无反馈的操作 == 没操作
```

### 23.5 松耦合
```
模块之间独立，一个改了不影响其他。

例子:
  - 采集脚本和数据存储分离
  - Dashboard 和数据库分离
  - Skill 之间互不依赖
  - 每个 Skill 可单独加载/卸载
```

### 23.6 可观测性
```
系统所有关键状态要能观测。

要求:
  - 日志: 每个操作留有痕迹
  - 指标: 关键数据可量化
  - 告警: 异常能通知到人
  - 追踪: 问题能回溯到源头
```
