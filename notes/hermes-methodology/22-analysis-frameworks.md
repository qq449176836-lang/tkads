## 二十二、分析框架

| 框架 | 核心 | 用途 | 示例 |
|---|---|---|---|
| **MECE** | 相互独立，完全穷尽 | 拆解问题 | 错误分类、需求拆解 |
| **金字塔** | 结论先行，以上统下 | 组织输出 | 报告、建议、分析 |
| **5W2H** | What/Why/Who/When/Where/How/HowMuch | 理解需求 | 接到新任务时 | 
| **PDCA** | Plan→Do→Check→Act | 执行闭环 | 每次任务即一次PDCA |
| **STAR** | Situation/Task/Action/Result | 汇报问题 | 错误报告、复盘总结 |

### 22.1 MECE 原则
```
相互独立 (Mutually Exclusive): 分类之间不重叠
完全穷尽 (Collectively Exhaustive): 覆盖所有可能性

例子 - 错误分类:
  ✅ 工具失败 | 数据不一致 | 权限不足 | 环境缺失 | 逻辑错误
  → 不重叠，覆盖所有类型

  ❌ 小问题 | 中等 | 大问题
  → 边界模糊，不MECE
```

### 22.2 金字塔原理
```
结论先行 → 一级论据 → 二级论据 → 细节

组织任何输出前，先问自己:
  "用户最需要知道的是什么？" → 放第一句
  "支撑这个结论最重要的3条是什么？" → 接着放
  "细节放在最后"
```

### 22.3 5W2H 需求理解框架
```
接到新需求时快速过一遍:

What:   用户到底想要什么？
Why:    为什么要这个？
Who:    谁会使用？谁受影响？
When:   什么时候要？截止时间？
Where:  在哪里运行？影响范围？
How:    打算怎么做？
HowMuch: 预算/资源/时间约束？
```

### 22.4 PDCA 执行循环
```
Plan:   计划（检索→评估→方案）
Do:     执行（按方案操作）
Check:  检查（验证结果）
Act:    改进（复盘→沉淀→调整）

每次任务都至少跑完一圈 PDCA。
```

### 22.5 STAR 汇报结构
```
Situation: 背景/环境
Task:      需要解决的问题
Action:    做了什么
Result:    结果如何

范例:
  S: 数据库导入后SKU数据少了9行
  T: 定位并修复数据缺失
  A: 检查约束设计→发现唯一约束不完整→修改为三联约束
  R: 18个SKU全部导入成功，验证通过
```
