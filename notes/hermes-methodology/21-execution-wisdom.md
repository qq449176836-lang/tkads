## 二十一、执行智慧

### 21.1 最小可行输出
```
先给一个能用的结果，再迭代优化。
不要追求第一次就完美。

例子:
  创建仓库 → 先上传 README，再慢慢补充
  写脚本 → 先跑通核心路径，再处理边界情况
```

### 21.2 幂等性原则
```
跑一次和跑十次结果一样，不产生副作用。

做法:
  - 所有数据插入用 ON CONFLICT DO NOTHING
  - 文件写入先检查是否存在
  - API 调用考虑重复调用场景
```

### 21.3 回滚能力
```
每次变更都要有撤销方案。

做法:
  - 改数据库前先备份
  - 改配置前先记录原值
  - 改文件前先保存原版
  - 每次变更都是一个可逆的操作
```

### 21.4 渐进交付
```
分批给结果，每一批都是可验证的。

做法:
  - 第一批: 核心功能
  - 第二批: 错误处理
  - 第三批: 优化项
  每批都先让用户确认再继续
```

### 21.5 防御性操作
```
不知道路径就先查，不知道命令就问。

原则:
  - 不明路径不操作
  - 不明命令不执行
  - 不明配置不修改
  - 先验证再操作，先测试再上线
```
